<b>"개략적인 규모 추정(back-of-the-envelope estimation)"</b>은 보편적으로 통용되는 성능 수치상에서 사고 실험(thought exeriments)을 행하여 추정치를 계산하는 행위로서, <b>어떤 설계가 요구사항에 부합할 것인지를 보기 위한 것</b>이다.

개략적인 규모 추정을 효과적으로 하려면 규모 확장성을 표현하는 데 필요한 기본기에 능숙해야 한다. (2의 제곱수, 응답지연(latency) 값, 가용성 관련 수치 등)

## 2의 제곱수

분산 시스템에서 다루는 데이터 양은 엄청나게 커질 수 있으나 그 계산법은 기본을 크게 벗어나지 않으며, 제대로 된 계산 결과를 얻으려면 <b>데이터 볼륨의 단위를 2의 제곱수로 표현</b>하면 어떻게 되는지를 알아야 한다.

- 최소 단위 :  1byte = 8bit
- ASCII 문자 하나가 차지하는 메모리 크기 : 1byte


|        2의 x 제곱        |       근사치       |        이름        | 축약형 |
| :----------------------: | :----------------: | :----------------: | :----: |
| 10 | 1천(thousand) | 1킬로바이트(Kilobyte) | 1KB |
| 20 | 1백만(million) | 1메가바이트(Megabyte) | 2MB |
| 30 | 10억(billion) | 1기가바이트(Gigabyte) | 1GB |
| 40 | 1조(trillion) | 1테라바이트(Terabyte) | 1TB |
| 50 | 1000조(quadrillion) | 1페타바이트(Petabyte) | 1PB |

### 핵심 인사이트

- 메모리는 빠르지만 디스크는 여전히 느리다.
- 디스크 탐색(seek)은 가능한 한 피하라
- 단순한 압축 알고리즘은 빠르다 → 네트워크 전송 전 압축 권장
- 데이터 센터 간 통신에는 시간이 걸린다 → 지역(region) 분산 설계 시 고려 필요

## 가용성에 관계된 수치들

- 고가용성(high availability) : 시스템이 오랜 시간 동안 지속적으로 중단 없이 운영될 수 있는 능력 (퍼센트로 표현하며 대부분 99% ~ 100%)
- SLA(Service Level Agreement) : 서비스 사업자(service provider)와 고객 사이에 맺어진 합의 (서비스 사업자가 제공하는 서비스의 가용시간이 공식적으로 기술되어 있음)
- 가용시간(uptime) : 관습적으로 숫자 9를 사용해 표시하며, 9가 많으면 많을수로 좋다.

### 9의 개수와 시스템 장애 시간(downtime) 사이의 관계

| 가용률 | 하루당 장애시간 | 주당 장애시간 | 개월당 장애시간 | 연간 장애시간 |
|--------|---------------|-------------|--------------|-------------|
| 99% | 14.40분 | 1.68시간 | 7.31시간 | 3.65일 |
| 99.9% | 1.44분 | 10.08분 | 43.83분 | 8.77시간 |
| 99.99% | 8.64초 | 1.01분 | 4.38분 | 52.60분 |
| 99.999% | 864.00밀리초 | 6.05초 | 26.30초 | 5.26분 |
| 99.9999% | 86.40밀리초 | 604.80밀리초 | 2.63초 | 31.56초 |


- 9가 하나 늘 때마다 장애 허용 시간이 10배 줄어든다 => 99% → 99.9%만 해도 연간 3.65일 → 8.77시간으로 급감
- 99% (two-nines)는 연간 3.65일 다운 허용 - 일반 서비스 수준
- 99.99% (four-nines)부터 연간 52분대 - 대부분의 상용 SLA 목표치
- 99.999% (five-nines)는 통신/금융 등 고가용성 시스템의 목표 기준
- 가용률이 높을수록 아키텍처 복잡도와 비용이 기하급수적으로 증가하므로, 비즈니스 요구사항에 맞는 현실적인 목표 설정이 중요하다

## 예제: 트위터 QPS와 저장소 요구량 추정

가정

- 월간 능동 사용자(monthly active user)는 3억 (300 million) 명 이댜
- 50%의 사용자가 트위터를 매일 사용한다.
- 평균적으로 각 사용자는 매일 2건의 트윗을 올린다.
- 미디어를 포함하는 트윗은 10% 정도다.
- 데이터는 5년간 보관된다.

```
[QPS(Query Per Second) 추정치]
- 일간 능동 사용자(Daily Active User, DAU) = 3억 x 50% = 1.5억(150million)
- QPS = 1.5억 x 2트윗 / 24시간 / 3600초 = 약 3500
- 최대 QPS(Peek QPS) = 2 x QPS = 약 7000

[미디어 저장을 위한 저장소 요구량]
- 평균 트윗 크기 = tweet_id(64 byte), text(140 byte), media(1 MB)
- 미디어 저장소 요구량 = 1.5억 x 2 x 10% x 1MB = 30TB/일
- 5년간 미디어를 보관하기 위한 저장소 요구량 = 30TB x 365 x 5 = 약 55 PB
```

## 팁

- 결과보다는 결과를 도출해 나가는 과정이 중요함
- 근사치를 활용한 계산(rounding and approximation) : 계산 결과의 정확함을 평가하는 것이 아니므로 적절한 근사치를 활용하여 시간을 절약할 것 ("99987/9.1"과 같은 복잡한 계산은 "100000/10"으로 간소화)
- 나중에 살펴볼 수 있도록 가정(assumption)들은 적어두기
- 스스로 햇갈릴 수 있으니, 단위(unit) 붙이기
